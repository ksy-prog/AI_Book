# **Критерии приемки MVP приложения «Книга AI»**

**1\. Введение**

Настоящий документ определяет критерии успешной приемки минимально жизнеспособного продукта десктопного приложения «Книга AI» на платформе Unity. Критерии разработаны на основе проектной документации и фиксируют набор проверок, которым должна удовлетворять программа.

Приемка осуществляется путем проведения серии тестовых прогонов в соответствии с процедурой, описанной в разделе 4. Каждый пункт проверяется отдельно и считается выполненным при полном совпадении фактического поведения системы с ожидаемым.

**2\. Объект приемки**

Объектом приемки является сборка десктопного приложения «Книга AI» под управлением Windows 10/11, включающая:

- исполняемый файл,
- локальный сервер Ollama с загруженной моделью saiga_llama3:latest,
- файлы моделей Vosk, размещённые в папке StreamingAssets,
- JSON-сценарий чтения,
- набор спрайтов персонажа.

Приложение должно обеспечивать полный цикл голосового сопровождения чтения без доступа в Интернет (за исключением локального HTTP-соединения с Ollama).

**3\. Критерии приемки**

3.1. **Функциональные требования**

3.1.1. Запуск и инициализация

- Приложение запускается без критических ошибок компиляции и выполнения.
- Все сервисы (MicrophoneService, SilenceDetector, VoskService, LLMService, TTSService) инициализируются автоматически при старте сцены.
- В консоль (или лог-файл) выводятся информационные сообщения о статусе инициализации каждого сервиса.

3.1.2. Конечный автомат состояний

- Начальное состояние системы – Listening.
- При обнаружении речи (громкость выше заданного порога) система переходит из Listening в Recording.
- При отсутствии речи в течение 7 секунд в состоянии Recording система автоматически останавливает запись и переходит в Processing.
- В состоянии Processing аудиофайл передаётся на распознавание (Vosk). После получения текста вызывается генерация ответа (LLM), затем система переходит в Speaking.
- После завершения озвучивания ответа система возвращается в Listening.
- При отсутствии речи в течение 5 секунд в состоянии Listening система переходит в PausedHelp.
- Из состояния PausedHelp через 2 секунды (или при возобновлении речи) система возвращается в Listening.
- Запрещены переходы, не предусмотренные диаграммой состояний (попытки должны логироваться как предупреждения).

3.1.3. Работа с микрофоном

- При старте выбирается первый доступный микрофон, его имя выводится в лог.
- Запись начинается корректно, AudioClip заполняется аудиоданными.
- Запись останавливается по команде, микрофон освобождается.
- Метод GetAudioClip() возвращает валидный клип после записи.

3.1.4. Детекция речи и тишины

- В состояниях Listening/Recording текущая громкость вычисляется и сравнивается с заданным порогом.
- При громкости выше порога таймер тишины сбрасывается, состояние меняется на Recording.
- При громкости ниже порога таймер тишины накапливается. По истечении 7 секунд в Recording вызывается переход в Processing. По истечении 5 секунд в Listening – переход в PausedHelp.
- В консоль выводится отладочная информация о громкости и таймерах не реже одного раза в 30 кадров.

3.1.5. Распознавание речи (STT)

- Аудиоклип успешно передаётся в Vosk, результатом является текстовая строка в кодировке UTF-8.
- Распознанный текст передаётся в SessionManager, а затем в SessionFlowController (при активной сессии) или в LLMService (в свободном режиме).
- В случае ошибки распознавания система возвращается в Listening без зависания.

3.1.6. Генерация ответа (LLM)

- Ollama сервер доступен по адресу http://localhost:11434.
- Запрос к модели saiga_llama3:latest формируется корректно (системный промпт дружелюбного персонажа, сообщение пользователя).
- Ответ содержит связный текст длиной 1–2 предложения, соответствующий роли персонажа и цели поощрения чтения.
- В случае ошибки соединения или таймаута (более 120 секунд) система выдаёт запасную фразу и не завершается аварийно.
- Ответ передаётся в TTSService для озвучивания.

3.1.7. Синтез речи (TTS)

- Текст ответа воспроизводится через синтезатор речи (System.Speech / SpeechLib) с использованием доступного русского голоса.
- Звук воспроизводится через AudioSource на том же GameObject.
- По окончании воспроизведения вызывается метод OnTtsFinished() в SessionManager, что приводит к возврату в Listening.

3.1.8. Сценарный движок (SessionFlowController)

- JSON-файл сценария (session_01.json) загружается без ошибок, структура соответствует утверждённой схеме.
- После загрузки сценария отрабатывает Preroll – произносится приветственная фраза.
- Сессия проходит последовательно по всем textBlocks:
    - после «чтения» пользователем (имитация паузой или нажатием кнопки) задаётся вопрос из массива questions;
    - после каждого вопроса система ожидает ответа пользователя (речь или нажатие «Далее»);
    - вопросы задаются до тех пор, пока не закончатся для текущего блока.
- После последнего блока произносится завершающая фраза (closing) и сессия завершается.
- Допускается ручное переключение вопросов/блоков с помощью UI-кнопок. При отсутствии UI смена происходит автоматически по паузам.

3.1.9. Пользовательский интерфейс (базовый)

- На основном Canvas присутствует индикатор состояния микрофона (MicIndicator), изменяющий цвет/иконку при записи.
- Имеется контейнер VisualRoot для размещения персонажа.
- (Опционально) Реализованы экраны: выбор книги, чтение, диалог, завершение – с возможностью навигации по кнопкам.
- Все элементы UI корректно отображаются при разрешении 1920×1080 и масштабируются.

3.1.10. Визуальное представление персонажа

- Спрайты Idle.png, Listening.png, Speaking.png присутствуют в Assets/Art и имеют тип Sprite (2D and UI).
- При старте отображается Idle спрайт.
- При переходе в Listening спрайт меняется на Listening.
- При переходе в Speaking спрайт меняется на Speaking.
- При возврате в Listening/Idle спрайт возвращается к соответствующему состоянию.
- Смена спрайтов не вызывает мерцания или зависаний.

3.2. **Нефункциональные требования**

3.2.1. Надёжность и устойчивость

- Полный цикл сценария (загрузка, приветствие, не менее 2 блоков с вопросами, завершение) выполняется без ручного вмешательства не менее чем в 80% тестовых прогонов (4 из 5).
- При возникновении исключений в любом из сервисов система не завершается аварийно, а возвращается в Listening с записью ошибки в лог.
- Отсутствуют утечки памяти при длительной работе (потребление RAM стабильно в течение 10 минут активного использования).

3.2.2. Производительность

- Средняя задержка между окончанием речи пользователя и началом озвучивания ответа не превышает 15 секунд (при условии, что модель Ollama уже загружена).
- Частота кадров в сцене не падает ниже 30 FPS при работающем микрофоне и TTS.

3.2.3. Безопасность и конфиденциальность

- В процессе работы приложение не отправляет аудио- или текстовые данные на внешние серверы (единственное сетевое взаимодействие – локальный HTTP-запрос к Ollama на 127.0.0.1:11434).
- API-ключи или иные секреты не хранятся в открытом виде в репозитория.

3.2.4. Совместимость и развёртывание

- Приложение стабильно работает под управлением Windows 10/11 x64.
- Для запуска не требуется предустановленных программ, кроме самой Unity-сборки и Ollama.
- Модели Vosk корректно загружается из папки StreamingAssets.

3.3. Педагогические критерии

- Ответы ассистента носят поощряющий и поддерживающий характер, не содержат негативных оценок.
- Ассистент не даёт прямых ответов на вопросы, а стимулирует ребёнка к самостоятельному размышлению (соответствует системному промпту).
- Вопросы, задаваемые ассистентом, соответствуют текущему текстовому блоку сценария и не нарушают последовательность.
- При длительной паузе (5 секунд) ассистент произносит мягкую подсказку, мотивирующую продолжить чтение.

3.4. Документация и сопровождающие материалы

- В корне репозитория присутствует файл README.md, содержащий:
    - краткое описание проекта,
    - инструкцию по установке и запуску,
    - структуру проекта,
    - описание используемых технологий.
- Код снабжён комментариями в ключевых методах.
- Приложен пример JSON-сценария с пояснениями.

4\. Процедура проверки

1.  Подготовка окружения:
    - Убедиться, что Ollama запущена, модель saiga_llama3:latest загружена.
    - Проверить доступность микрофона.
    - Запустить приложение (из Unity Editor или собранный .exe).
2.  Проверка инициализации:
    - В консоли должны появиться сообщения об успешной инициализации всех сервисов.
3.  Проверка голосового цикла:
    - Произнести короткую фразу, убедиться в переходе Listening → Recording → Processing → Speaking → Listening.
    - Оценить качество ответа (осмысленность, соответствие роли).
    - Промолчать 5 секунд – переход в PausedHelp, затем автоматический возврат.
4.  Проверка сценарного режима:
    - Запустить сценарий (через UI или автоматически).
    - Проследить полный путь по блокам и вопросам.
    - Убедиться в корректном завершении.
5.  Проверка визуальной части:
    - Индикатор микрофона меняется при записи.
    - Спрайт персонажа соответствует состоянию.
6.  Повторить пункты 3–4 не менее 5 раз для оценки устойчивости (критерий из п. 3.2.1).
7.  Проверить документацию и соответствие критериям раздела 3.4.
